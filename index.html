Bài 1

Agent Core Loop - Bộ não Tư duy của AI

Phân tích sâu sắc về ReAct pattern (Reasoning & Acting), vòng lặp Think-Act-Observe, và cơ chế đưa ra quyết định tự động của Agent.

1. Agent Thực Sự Khác Biệt Thế Nào?

Chào các bạn. Rất nhiều người nhầm lẫn giữa một LLM thông thường (như ChatGPT ban đầu) và một AI Agent. Bản chất của một Chatbot là một mô hình sinh văn bản tĩnh: bạn cung cấp text đầu vào, nó dự đoán text đầu ra. Nó không có "tay chân", không thể nhận thức thế giới bên ngoài ngoài những gì bạn vừa gõ.

Trong khi đó, một Agent là một thực thể có khả năng tự chủ. Nó nhận một mục tiêu phức tạp, tự phân rã vấn đề, tự quyết định gọi các Tools (công cụ như đọc file, chạy terminal, search web), quan sát kết quả trả về từ môi trường, và tiếp tục suy luận cho đến khi hoàn thành mục tiêu. Điểm khác biệt cốt lõi: Agent sở hữu một Vòng lặp Nhận thức và Hành động (Cognitive Loop).

2. Bản Chất Của ReAct (Reason + Act) Loop

Trái tim của mọi Agent hiện đại là ReAct pattern. Thay vì bắt LLM nghĩ ra toàn bộ giải pháp ngay lập tức, ReAct buộc LLM phải đan xen giữa Suy luận (Reasoning) và Hành động (Acting). Dưới đây là cách luồng thông tin di chuyển:

flowchart TD User([User Request]) --> A A[THINK\nLLM phân tích vấn đề và lập kế hoạch] --> B{Cần dùng Tool?} B -- Có --> C[ACT\nGọi Tool / Hàm cụ thể] C --> D[OBSERVE\nNhận kết quả từ môi trường] D --> A B -- Không --> E([Final Answer\nHoàn thành nhiệm vụ]) style A fill:#21262d,stroke:#58a6ff,stroke-width:2px style C fill:#21262d,stroke:#3fb950,stroke-width:2px style D fill:#21262d,stroke:#d29922,stroke-width:2px

3. Phân Tích Trace Thực Tế: Sửa Một Bug

Hãy xem một Agent thực tế giải quyết yêu cầu "Fix bug trong login.py" như thế nào. Lưu ý cách nó phản hồi với các rào cản:

Vòng lặp (Iter)Quá Trình Suy Luận (Think)Hành Động (Act)Kết Quả Quan Sát (Observe)
1Tôi cần đọc mã nguồn của file login.py để hiểu logic hiện tại.read_file("login.py")[Trả về nội dung 200 dòng]
2Tôi thấy hàm xác thực chưa xử lý lỗi None. Tôi cần tìm xem có test case nào không.glob("**/test_login*.py")Tìm thấy tests/test_login.py
3Tôi sẽ vá hàm login bằng cách thêm check null ở dòng 42.edit("login.py", diff)"Patch applied successfully"
4Bây giờ tôi phải xác minh xem code tôi vừa sửa có làm hỏng gì không.bash("pytest tests/test_login.py")"1 failed, 4 passed" (Vẫn còn lỗi)
5Ah, test fail do thiếu import ValueError. Tôi sẽ thêm vào đầu file.edit("login.py", diff_import)"Patch applied successfully"
6Kiểm tra lại lần nữa.bash("pytest tests/test_login.py")"5 passed, 0 failed"
7Mọi thứ đã xanh. Tôi có thể báo cáo hoàn thành.Trả về Final Answer"Đã fix bug null check..."

Ở đây, agent đã tự chữa lành lỗi do chính nó gây ra (iteration 4-5). Đây chính là sức mạnh của vòng lặp Observe. Không có nó, code sinh ra sẽ chỉ là phỏng đoán mù quáng.

Bài Học Cốt Lõi (Takeaways)

  1. Mô hình Công thức: Agent = LLM + Tools + Control Loop. LLM là não bộ, Tools là tay chân, Control Loop là nhịp đập của sự sống.
  2. Chất lượng của một Agent không chỉ dựa vào mô hình to hay nhỏ, mà phụ thuộc cực lớn vào độ tin cậy của Tools và cơ chế xử lý lỗi (error handling) trong Loop.
  3. Chi phí (Cost): Mỗi vòng lặp (Iteration) tiêu tốn Token. Một Agent giỏi là Agent biết lập kế hoạch tốt để hoàn thành mục tiêu với số lần gọi tool ít nhất.
Bài 2

System Prompt Design - Kiến trúc "DNA" của Agent

System Prompt không đơn thuần là một đoạn text, nó là kiến trúc phần mềm quyết định toàn bộ giới hạn nhận thức và quy chuẩn hành vi.

1. Đừng Viết Prompt, Hãy "Thiết Kế" Prompt

Một sai lầm phổ biến khi lập trình viên bắt đầu làm Agent là viết System Prompt như viết một bài văn dài lê thê: "Bạn là AI giỏi, hãy làm việc này, đừng làm việc kia, nhớ dùng tool này nhé...". Khi độ phức tạp tăng lên, LLM sẽ bắt đầu quên các chỉ thị ở giữa, hoặc lẫn lộn các nguyên tắc. System Prompt của một Agent Production cần phải có tính cấu trúc sâu sắc, phân tầng rõ rệt hệt như mô hình OSI trong mạng máy tính.

2. Cấu Trúc 7 Tầng Chuyên Sâu

Chúng ta phân chia prompt thành 7 layer độc lập, được tiêm (inject) tuần tự. Điều này giúp dễ dàng mở rộng và gỡ lỗi (debug) hành vi của Agent.

block-beta columns 1 A["Layer 1: IDENTITY (Định danh, Vai trò cốt lõi)"] B["Layer 2: TONE & STYLE (Phong cách giao tiếp, Concise/Verbose)"] C["Layer 3: SAFETY GUARDRAILS (Ranh giới đỏ không được vượt qua)"] D["Layer 4: TOOL POLICIES (Luật sử dụng bộ công cụ)"] E["Layer 5: TASK GUIDELINES (Quy trình chuẩn - SOP để giải quyết vấn đề)"] F["Layer 6: DYNAMIC CONTEXT (OS, Dir, Git Status hiện tại)"] G["Layer 7: AVAILABLE SKILLS (Các plugin động đang được nạp)"]

3. Phân Tích Các Layer Quan Trọng

Layer 3: Safety Guardrails (Ranh giới đỏ)
Đây là nơi bạn bảo vệ hệ thống của mình khỏi sự ngớ ngẩn của AI. Ví dụ: "NEVER use `rm -rf`. ALWAYS ask user before executing git push. DO NOT modify files outside the workspace." LLM tuân thủ chỉ thị cấm (Negative Prompt) tốt nhất khi chúng được gom nhóm lại thành một khối cảnh báo mạnh mẽ.

Layer 5: Task Guidelines (Quy trình chuẩn - SOP)
Thay vì để LLM tự bơi, hãy dạy nó cách làm việc của một Senior Engineer. Ví dụ: "1. Explore the codebase using `grep` before writing code. 2. Write unit tests before implementing the logic. 3. Always run tests to verify."

Layer 6: Dynamic Context (Ngữ cảnh động)
System prompt không tĩnh. Trước khi gửi cho API, bạn phải "Bơm" thông tin thực thời vào. Ví dụ: Agent cần biết hệ điều hành đang dùng là Windows hay Linux để viết script bash cho đúng. Nó cần biết thư mục hiện tại (`pwd`). Mọi thứ được render qua Template Engine (như Jinja2).

Bài Học Cốt Lõi (Takeaways)

  1. Sử dụng Builder Pattern: Đừng dùng một chuỗi string khổng lồ. Hãy tạo một class SystemPromptBuilder để add từng layer vào một cách linh hoạt theo session.
  2. Position Bias: Thông tin ở đầu và cuối prompt được LLM ghi nhớ tốt nhất. Hãy đặt Identity ở đầu, và Dynamic Context/Rules ở sát cuối.
  3. Chi phí Token: System Prompt dài tốn rất nhiều tiền vì nó được gửi kèm trong MỌI VÒNG LẶP (Iteration). Tối ưu hóa câu chữ là bắt buộc.
Bài 3

Tool System - Đôi Tay Linh Hoạt Của AI

Phân tích chuyên sâu về Function Calling, thiết kế Tool chuẩn Unix Philosophy, và cơ chế thực thi song song (Parallel Execution).

1. Thiết Kế Tool Theo Triết Lý Unix

Nếu bạn thiết kế một tool tên là fix_code_and_run_test, bạn đã thất bại. Một tool tốt phải tuân thủ nguyên tắc: Làm một việc duy nhất, và làm nó thật tốt.

Agent cần các primitive tools (công cụ cơ sở) để tự tổ hợp chúng lại. Tại sao? Vì nếu tool quá phức tạp, LLM sẽ rất khó suy luận ra tham số (parameters) đúng cho nó, dẫn đến ảo giác (hallucination) trong việc gọi hàm.

Nhóm Chức NăngPrimitive Tools (Nên Dùng)Giải Thích Của Giảng Viên
File Systemread_file, write_file, edit_file_diff, globLuôn tách riêng việc đọc toàn bộ file và sửa đổi file bằng diff (giúp tiết kiệm token).
Terminalrun_bash, kill_processBash tool là quyền lực nhất, nhưng phải có timeout (vd: 120s) và tự động cắt ngắn (truncate) output nếu quá dài.
Code Intelsearch_code, find_usagesThay vì để LLM tự grep mù quáng, cung cấp tool Semantic Search bằng AST (Abstract Syntax Tree).

2. Parallel vs Sequential Execution (Thực thi Song Song)

Bí quyết để Agent chạy nhanh như chớp là khả năng gọi nhiều tool cùng lúc. LLM hiện đại hỗ trợ trả về nhiều tool_calls trong một response.

sequenceDiagram participant LLM as Agent (LLM) participant T1 as read_file(file_A) participant T2 as read_file(file_B) participant T3 as read_file(file_C) note over LLM: Agent phát hiện cần đọc 3 file độc lập LLM->>T1: [Parallel Call] LLM->>T2: [Parallel Call] LLM->>T3: [Parallel Call] T1-->>LLM: Output A T2-->>LLM: Output B T3-->>LLM: Output C note over LLM: Tiết kiệm được 2/3 thời gian so với gọi tuần tự!

Cạm bẫy: Không phải lúc nào cũng Parallel được. Nếu hành động sau phụ thuộc vào kết quả của hành động trước (ví dụ: tạo file xong mới chạy test), Agent phải được dạy cách phân biệt và gọi tuần tự.

3. Xử Lý Lỗi (Error Handling) Cho Tools

Đây là điểm khác biệt giữa code đồ chơi và production. Khi tool bash chạy lỗi (ví dụ: lệnh không tồn tại, hoặc python crash), tuyệt đối không được throw exception làm sập chương trình.

Bạn phải bắt (catch) toàn bộ lỗi, đóng gói nó thành một chuỗi văn bản thân thiện (ví dụ: "Command failed with exit code 1. Stderr: ...") và ném trả lại cho vòng lặp Observe. Chính LLM sẽ đọc cái lỗi đó và tự nói: "À, lỗi cú pháp, để mình sửa lại tham số truyền vào."

Bài Học Cốt Lõi (Takeaways)

  1. JSON Schema Định Hình Trí Thông Minh: LLM hiểu tool qua các field description trong schema. Mô tả tham số càng chi tiết, LLM gọi tool càng chuẩn xác.
  2. Graceful Degradation: Tools không bao giờ crash app chính. Mọi lỗi của môi trường là thông tin quý giá để Agent tự học và sửa chữa.
  3. Diff Tool là Chân Ái: Đối với file lớn (1000 dòng), việc bắt LLM viết lại (rewrite) cả file là thảm họa chi phí. Hãy bắt nó gửi chuẩn Git Merge Diff hoặc Unified Diff.
Bài 4

Context Management - Quản Trị Trí Nhớ Giới Hạn

Giải quyết bài toán muôn thuở: Khi lịch sử chat quá dài, Token bị đầy, và LLM bắt đầu "mất trí nhớ".

1. Khủng Hoảng Context Window

Giả sử bạn có context window là 128K tokens. Nghe có vẻ nhiều, nhưng khi Agent liên tục chạy lệnh bash, grep kết quả (đôi khi output lên tới vài chục nghìn dòng), và đọc các file lớn, chỉ sau khoảng 10-15 vòng lặp, context của bạn sẽ bị đầy. Khi đó, API từ chối phục vụ (Error: Token limit exceeded) hoặc mô hình bắt đầu bị "Attention dilution" (suy giảm tập trung, quên mất mục tiêu ban đầu).

2. Cơ Chế Auto-Summarization (Tóm tắt tự động)

Giải pháp hiệu quả nhất là Cuộn Lịch Sử (Rolling History) kết hợp Tóm tắt.

  • Giữ lại nguyên bản: K system prompt (không bao giờ đổi) + N tin nhắn mới nhất (giữ chi tiết sắc nét).
  • Nén phần giữa: M tin nhắn cũ ở giữa được gửi cho một cuộc gọi LLM nhỏ hơn, giá rẻ (như GPT-4o-mini) để tóm tắt thành một đoạn văn bản ngắn: "Tóm tắt: Agent đã đọc file X, chạy lệnh Y, tìm thấy lỗi Z và đang lên kế hoạch vá."
  • Sau đó chèn đoạn tóm tắt này vào giữa System prompt và các tin nhắn mới.

3. Hệ Thống Bộ Nhớ Dài Hạn (Long-term Memory)

Auto-summarization chỉ giải quyết trí nhớ trong 1 session làm việc. Nhưng nếu ngày hôm sau bạn lại bảo Agent sửa tính năng đó thì sao? Đây là lúc Vector Database và Kiến trúc Memory xuất hiện.

flowchart LR Chat[Current Session] --> |Extract Facts| E[Knowledge Extractor Agent] E --> |Store| DB[(Vector / Graph DB)] New[New Session] --> |Search Context| DB DB --> |Inject Relevant Facts| Prompt[System Prompt]

Agent được cung cấp các tool đặc biệt: save_memory, search_memory. Trước khi kết thúc một task lớn, nó có nhiệm vụ ghi lại vào não bộ: "Quy ước code của dự án này dùng camelCase, database là Postgres, lệnh start là 'pnpm run dev'." Lần sau, nó tự động search và cấu hình tư duy ngay từ đầu.

Bài Học Cốt Lõi (Takeaways)

  1. Context là Bất động sản đắt đỏ: Mọi output từ Terminal hay File content phải được Truncate (Cắt bớt) nếu nó vượt quá một ngưỡng nhất định (ví dụ 3000 kí tự). Đừng bao giờ nhét toàn bộ output của lệnh npm install vào não LLM.
  2. Sự tập trung: Khi context quá dài, mô hình xịn đến mấy cũng sẽ xao nhãng. Tóm tắt giúp tái tập trung sự chú ý của Agent vào mục tiêu cốt lõi.
Bài 5

Multi-Agent Orchestration - Phân Chia Trị Vì

Từ một Agent cô độc vươn lên thành một cấu trúc Tổ chức đa Đặc vụ, nơi mỗi Agent đóng một vai trò chuyên biệt.

1. Tại Sao Lại Cần Nhiều Agent?

Nếu bạn dồn 25 tools khác nhau (từ lướt web, sửa code, đến thiết kế UI) vào cho một Agent duy nhất, nó sẽ bị "ngợp". System prompt sẽ phình to ra khổng lồ, và trong quá trình suy luận, nó phải lựa chọn giữa quá nhiều chức năng dẫn đến việc đưa ra quyết định sai lầm. Giải pháp là Separation of Concerns (Phân tách trách nhiệm).

2. Kiến Trúc Router / Orchestrator

Chúng ta xây dựng mô hình có một "Quản lý trưởng" (Orchestrator) và nhiều "Chuyên viên" (Sub-agents).

flowchart TD User([User Request]) --> Router["Orchestrator Agent\n(Phân tích & Điều phối)"] Router -- "Ủy quyền Task A" --> Planner["Plan Agent\n(Chuyên gia Phân tích & Lập Kế Hoạch)"] Router -- "Ủy quyền Task B" --> Explorer["Explore Agent\n(Chuyên gia Tìm kiếm Codebase)"] Router -- "Ủy quyền Task C" --> Coder["Code Agent\n(Chuyên gia Viết & Sửa Code)"] Router -- "Ủy quyền Task D" --> Reviewer["Review Agent\n(Chuyên gia Kiểm thử & Audit)"] Planner -.-> Router Explorer -.-> Router Coder -.-> Router Reviewer -.-> Router style Router fill:#1f6feb,stroke:#58a6ff,stroke-width:2px,color:#fff

3. Vòng Đời Của Việc Ủy Quyền (Delegation)

Khi Orchestrator cần tìm hiểu kiến trúc dự án, nó sẽ gọi hàm: delegate_to_explorer(query="Find how database connection is initialized").

Điều kỳ diệu xảy ra ở đây: Explore Agent được sinh ra (spawn) trong một Context window hoàn toàn MỚI, sạch sẽ. Nó có system prompt riêng (chỉ tập trung vào grep và read file). Nó thực thi một vòng lặp ReAct độc lập. Sau 5 vòng lặp, nó tìm ra câu trả lời và trả về Kết quả tóm tắt cho Orchestrator.

Orchestrator KHÔNG BỊ Ô NHIỄM bởi toàn bộ các file rác mà Explore Agent đã đọc. Nó chỉ nhận được tinh hoa kiến thức. Đây là cách giải quyết triệt để bài toán Context Window ở Bài 4.

Bài Học Cốt Lõi (Takeaways)

  1. Chuyên biệt hóa System Prompt: Mỗi Sub-agent có một System prompt cực kỳ tối ưu cho đúng vai trò của nó. Code Agent không cần biết tool lướt web. Browser Agent không cần biết lệnh git.
  2. Lớp bảo vệ Context (Context Isolation): Việc gọi Sub-agent biến một quá trình dài dòng thành một lời gọi hàm hộp đen (blackbox function call), bảo vệ trí nhớ của Orchestrator.
Bài 6

Skills & MCP - Giao Thức Mở Rộng Hệ Sinh Thái

Cách tích hợp các ứng dụng bên thứ ba (Database, GitHub, Figma) vào Agent mà không cần sửa đổi mã nguồn lõi.

1. Vấn Đề Của Hard-coded Tools

Ở Bài 3, chúng ta định nghĩa các công cụ cốt lõi (Core tools) như file system, bash. Nhưng nếu công ty bạn muốn Agent truy vấn trực tiếp vào Database nội bộ, hoặc kéo issue từ Jira? Bạn không thể cứ liên tục thêm code vào repo của Agent được. Hệ thống sẽ trở thành một đống bùng nhùng (Monolith).

2. Giao thức MCP (Model Context Protocol)

MCP do Anthropic đề xuất là một bước ngoặt kiến trúc. Nó giống như chuẩn kết nối USB Type-C cho AI Agent. Thay vì viết mã tích hợp trực tiếp, bạn chạy các MCP Server độc lập (có thể bằng NodeJS, Python, Go).

flowchart LR Agent["Core AI Agent\n(Client)"] <--> |"JSON-RPC over stdio/HTTP"| Server1["Postgres MCP Server"] Agent <--> |"JSON-RPC"| Server2["GitHub MCP Server"] Agent <--> |"JSON-RPC"| Server3["Jira MCP Server"] subgraph External World Server1 --> DB[("(Database)")] Server2 --> GH["GitHub API"] Server3 --> Jira["Jira API"] end

Cơ chế: Khi Agent khởi động, nó ping các MCP Server đang chạy. Server phản hồi lại: "Tôi cung cấp tool X và Y, kèm theo Schema này". Agent tự động tiêm các công cụ này vào cấu hình (Dynamic Injection). Đối với Agent, nó không hề biết Postgres là gì, nó chỉ thấy có một Tool khả dụng và nó dùng.

3. Kiến Trúc "Skills" Bằng Prompt Động

Bên cạnh MCP, Agent còn hỗ trợ cấu trúc Skill (Kỹ năng mềm). Thay vì viết code Python cho Tool, bạn định nghĩa một Skill bằng... một file Markdown (`SKILL.md`).

Ví dụ: Kỹ năng deploy_vercel. Khi người dùng yêu cầu "Hãy deploy dự án này", Agent quét thư mục, thấy có kỹ năng này. Nó tự động đọc file Markdown chứa hướng dẫn từng bước của người kỹ sư thật: "Bước 1: check package.json, Bước 2: chạy lệnh vercel build, Bước 3: lấy link URL". Và Agent làm theo hệt như một bản hướng dẫn (SOP). Đây là cách tuyệt vời để tùy chỉnh Agent cho từng dự án cụ thể mà không cần động vào Core LLM.

Bài Học Cốt Lõi (Takeaways)

  1. Decoupling (Giảm kết dính): Tách biệt khả năng cốt lõi (Core Engine) khỏi các công cụ domain-specific thông qua giao thức chuẩn hóa (MCP).
  2. Empower Users: Bằng cách dùng Prompt-based Skills, bất kỳ người dùng nào cũng có thể "Dạy" thêm kỹ năng cho Agent chỉ bằng cách viết file văn bản thông thường.
Bài 7

Guardrails & Production - Đưa Agent Lên Chuyến Tàu Thực Tế

Xây dựng hàng rào an toàn, hệ thống đánh giá (Evaluation) và các biện pháp bảo vệ để Agent không phá nát máy chủ của bạn.

1. Kỹ Thuật Đảm Bảo An Toàn (Guardrails)

Một Agent có quyền thực thi lệnh Bash trên máy chủ của bạn là một rủi ro bảo mật khổng lồ. Chúng ta cần thiết lập hàng rào 3 lớp khắt khe:

  • Input Guardrails (Lọc Đầu Vào): Kiểm tra yêu cầu của User có chứa ý đồ xấu (Prompt Injection, Jailbreak) không trước khi cho Agent xử lý.
  • Execution Guardrails (Khóa Thực Thi): Các công cụ nhạy cảm (như run_bash) chạy trong một Sandbox (Docker container hoặc restricted shell). Cấm truy cập các file ngoài thư mục dự án. Bắt buộc Agent gọi tool AskUserConfirmation trước khi thực hiện các lệnh nguy hiểm như xóa file hàng loạt, git push, thao tác cơ sở dữ liệu.
  • Output Guardrails (Lọc Đầu Ra): Không cho phép in trực tiếp mật khẩu, API Key hay thông tin nhạy cảm vô tình đọc được ra màn hình cho user xem.

2. Continuous Evaluation (Đánh giá liên tục)

Bạn đổi một dòng trong System Prompt, làm sao bạn biết Agent không bị "ngu" đi ở một kỹ năng khác? Đó là lý do bạn cần một Eval Pipeline.

Thay vì Unit Test truyền thống (có output cố định), Test cho AI cần dùng một LLM khác để làm "Giám khảo" (LLM-as-a-Judge). Quá trình chạy CI/CD sẽ cấp cho Agent 50 bài toán thực tế đã biết trước kết quả (ví dụ: Fix bug X trong repo Y). Sau khi chạy tự động, Giám khảo sẽ kiểm tra xem diff tạo ra có giải quyết đúng vấn đề không, code có tuân thủ linting không, số vòng lặp tốn kém bao nhiêu.

3. Observability & Monitoring (Giám sát)

Khi đưa lên Production, bạn phải theo dõi Agent giống như theo dõi một Microservice phức tạp:

  • Trace tracking: Ghi lại mọi Thought, Tool Call, Output vào một hệ thống log (như LangSmith, Phoenix). Nếu Agent thất bại, bạn có thể xem lại đoạn băng (replay) quá trình suy luận để debug tại sao nó rẽ nhánh sai.
  • Cost monitoring: Tracking chính xác bao nhiêu Input Tokens, Output Tokens đã được dùng trong mỗi Session. Cảnh báo nếu một phiên làm việc có dấu hiệu lặp vô hạn (Infinite Loop).

Bài Học Cốt Lõi (Takeaways)

  1. An Toàn là Ưu Tiên Số 1: Đừng bao giờ trao quyền truy cập gốc (Root) hoặc quyền sinh tử cho Agent mà không có sự giám sát (Human-in-the-loop).
  2. LLMs Không Cố Định: Các nhà cung cấp model (OpenAI, Anthropic) thường xuyên cập nhật model âm thầm, làm thay đổi hành vi Agent. Eval Pipeline là vũ khí duy nhất chống lại sự suy thoái này.
  3. Kỷ nguyên của Agent: Lập trình viên tương lai không chỉ viết code, mà thiết kế môi trường, công cụ, và quy chuẩn để các AI Agent tự viết code.
